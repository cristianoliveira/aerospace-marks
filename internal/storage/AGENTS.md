# internal/storage/

Persistence layer. SQLite database holding marks, with goose migrations and
sqlc-generated queries.

## Responsibility

- `MarkStorage` interface (`marks.go`) — the domain contract every command
  programs against: add, get, toggle, replace, delete marks, close, and
  expose the underlying client.
- `MarkStorageClient` — production implementation composing sqlc `queries.Queries`
  over a `StorageDBClient`.
- `StorageDBClient` interface + `StorageClient` (`client.go`) — wraps
  `*sql.DB`, owns connection lifecycle.
- `MarksDatabaseConnector` — connects, runs migrations, returns a
  `StorageDBClient`. Instantiated in `main.go`.
- Migrations (`db/migrations/*.sql`) run via goose on connect (embedded FS).

DB path defaults to `$HOME/.local/state/aerospace-marks/storage.db`; override
with `AEROSPACE_MARKS_DB_PATH` (see `internal/constants`).

## sqlc generated code — DO NOT EDIT

`db/queries/marks.sql.go`, `models.go`, `db.go` are generated by sqlc from:
- schema: `db/migrations`
- queries: `db/queries/marks.sql` (hand-authored)
- config: `sqlc.yml`

To change a query: edit `db/queries/marks.sql`, then regenerate with sqlc.
Add a new migration under `db/migrations/` for schema changes.

## What belongs here

- New mark operations on `MarkStorage` (add a method to the interface +
  implementation + a sqlc query + a migration if the schema changes).
- Connection/config concerns.

## What does NOT belong here

- Window lookup logic (that is `internal/aerospace`).
- Output formatting (that is `internal/format`).
- CLI concerns (that is `cmd`).

## Testing

- Production code is exercised through command tests (`cmd/*_test.go`) via the
  gomock `MockMarkStorage` (see `internal/mocks`).
- Mocks are regenerated with `scripts/mock-generator.sh` whenever interfaces
  here change.
